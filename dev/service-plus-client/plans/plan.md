# Opening Jobs: behavior & functionality changes

Source spec: `plans/tran.md`. This plan covers all 7 bullet points, plus one confirmed extension
to bullet #6 (the "Opening" pill also appears in Final-a-Job and Deliver-Job grids, not just Job
Control/Pipeline). Investigated read-only across both `service-plus-client` and
`service-plus-server` тАФ no code changes made yet.

## Context

Opening Jobs is the feature for entering legacy/pre-existing jobs. Today it has three problems
that this plan fixes together, since they're related pieces of the same design:

1. The job number the user types is discarded тАФ Opening Jobs' create path already calls the same
   `createSingleJob` mutation Single Job uses, and that resolver **unconditionally overwrites**
   `job_no` with a server-generated sequence number regardless of what the client sends. So the
   `"Z-"`-prefixed value Opening Jobs currently computes client-side never actually reaches the
   DB тАФ it's silently thrown away. The user's typed number should instead be saved as
   `alternate_job_no` (exactly like Single Job already does for its own alt-reference field).
2. Because opening jobs live in the same `job` table as regular jobs, other grids (Single Job,
   Job Control, Job Pipeline) need to explicitly decide whether to include or exclude them, and
   currently none of them look at `is_opening_job` at all. Single Job should exclude them; Job
   Control and Job Pipeline should include them but visually flag them with an "Opening" pill тАФ
   which only makes sense once `is_opening_job` is actually queried and typed through.
3. The transaction-history "first entry" for every job (single or opening) is a client-invisible
   synthetic row тАФ job creation never inserts a real `job_transaction` row, so the query that
   powers the transaction summary fabricates one, hardcoded to say "Received". For opening jobs
   this should say "Opening Job" instead.

None of this requires new columns тАФ `job.is_opening_job` and `job.alternate_job_no` already
exist and are already used by Opening Jobs' create/edit payload (`is_opening_job: true`) and by
Single Job (`alternate_job_no`) respectively.

## Part A тАФ job_no auto-generated by server; user input тЖТ alternate_job_no (tran.md #2, #3)

**Root cause (confirmed):** `resolve_create_single_job_helper`
(`service-plus-server/app/graphql/resolvers/mutation_helper.py:1170-1234`) always claims a job
number via `SqlStore.CLAIM_NEXT_JOB_NUMBER` (`sql_store.py:832-838`, atomically increments the
branch's `JOB_SHEET` document_sequence counter) and does `x_data["job_no"] = job_no`
unconditionally before insert тАФ it never reads anything else from the payload for this purpose.
Single Job's create `sqlObject` (`single-job-section.tsx:196-218`) already omits `job_no`
entirely and relies on this. **No server change needed for auto-generation itself** тАФ it already
works exactly as required; Opening Jobs just needs to stop fighting it.

Also relevant: Single Job's **edit** path never sends `job_no` either, so `resolve_generic_update_helper` тЖТ `process_data`/`get_update_sql` (which build the `UPDATE` column
list purely from whatever keys are present in `x_data`) simply never touches that column on
update. Opening Jobs' edit path currently *does* resend `job_no`, which is a latent bug тАФ editing
an opening job can rewrite its job_no. Removing it fixes this for free.

### Client changes

**Important UX distinction (per user):** the form still presents a mandatory, user-typed field
*labeled* "Job No" тАФ that's the number the user actually cares about entering (their legacy/
external reference). It is **not** the DB `job_no` column though тАФ it saves to `alternate_job_no`.
The real DB `job_no` (server auto-generated, per Part A's root-cause finding) is shown as a
separate, secondary read-only field so its origin isn't confused with the user's input. Concretely
this means swapping the required/label roles of the two fields from the original draft of this
plan:

- **"Job No"** (required, red asterisk, free text) тЖТ maps to `alternate_job_no`. This is the field
  the user must fill in.
- **"System Job No"** (read-only, shows `"AUTO"` before save / the real generated `job.job_no`
  after save) тЖТ no longer user-editable, no longer required, just informational тАФ same styling
  Single Job already uses for its read-only auto field.

**`opening-job-schema.ts`**
- Remove the `job_no` field from `openingJobFormSchema` (it stops being a free-text form field).
- Add `alternate_job_no: z.string().min(1, MESSAGES.ERROR_OPENING_JOB_NO_REQUIRED)` тАФ **required**,
  reusing the existing `ERROR_OPENING_JOB_NO_REQUIRED` message (repurposed to validate this field
  instead of the old `job_no` field; do not remove the constant).
- Remove `job_no: ""` from `getOpeningJobDefaultValues()`; add `alternate_job_no: ""`.
- Delete the now-dead `normalizeJobNo()` function (only caller is removed below).

**`opening-job-form.tsx`**
- Drop `job_no: d.job_no,` from the edit-load `form.reset({...})` block (no longer a form field);
  add `alternate_job_no: d.alternate_job_no ?? "",`.
- Replace the old "Job No" `<Input {...register("job_no")}>` block (currently ~line 167-177) with
  a **read-only "System Job No" field**, mirroring Single Job's read-only pattern
  (`new-single-job-form.tsx:198-215`): `readOnly`, `value={editJob?.job_no ?? "AUTO"}`, same
  styling (`font-mono text-(--cl-accent) font-bold cursor-not-allowed opacity-80`), no required
  marker.
- Add a new **required "Job No"** input (red asterisk, `errors.alternate_job_no` тЖТ red border +
  message below, same pattern as the other required fields in this form e.g. Customer), registered
  to `alternate_job_no`, placeholder e.g. `"Enter the job numberтАж"`.

**`opening-job-section.tsx`**
- Remove `const normalizedJobNo = normalizeJobNo(values.job_no);` and the now-unused
  `normalizeJobNo` import.
- Edit branch xData (~line 302-326): remove `job_no: normalizedJobNo,`; add
  `alternate_job_no: values.alternate_job_no.trim(),`.
- Create branch `sqlObject` (~line 334-364): remove the top-level `doc_sequence_id: null,
  doc_sequence_next: null` keys (dead тАФ never read by the resolver) and `job_no: normalizedJobNo,`
  from xData; add `alternate_job_no: values.alternate_job_no.trim(),`.
- Leave the pre-save guard `if (!editJob && (!jobSheetSequence || !jobSheetSequence.prefix.trim()))`
  as-is тАФ it's now directly meaningful (blocks creation when the branch has no configured
  `JOB_SHEET` sequence, which `CLAIM_NEXT_JOB_NUMBER` depends on).

**`messages.ts`**
- No change тАФ `ERROR_OPENING_JOB_NO_REQUIRED` stays, now validating `alternate_job_no`.

**Consequence:** after this change, an opening job's real DB `job_no` is formatted identically to
a Single Job's (same `JOB_SHEET` sequence/prefix) and is secondary/read-only in the form. The
user-facing "Job No" the data-entry operator types is always required and always lands in
`alternate_job_no` тАФ which also means the Opening Jobs grid's existing "Alt: тАж" chip (added in
earlier work this session) will now always be populated for opening jobs, and Part D's "Opening"
pill becomes the primary way to distinguish these jobs by origin in Job Control/Pipeline/Final/
Deliver grids, since job_no format alone no longer carries a `"Z-"` visual cue.

### Not changed
- `GET_OPENING_JOBS_PAGED`/`COUNT` and `GET_JOB_DETAIL` (server): already select/search
  `alternate_job_no`. `JobControlRow`/`JobDetailType` client types: already have the field.

## Part E тАФ Transaction summary: first entry shows "Opening Job" (tran.md #7)

**Root cause (confirmed):** Job creation (`resolve_create_single_job_helper`) never inserts a
real `job_transaction` row for either Single Job or Opening Job. The transaction-history query
`GET_JOB_TRANSACTIONS_BY_JOB` (`sql_store.py:4411-4458`) is a `UNION ALL` of real
`job_transaction` rows plus a **synthesized fallback row** (`id = 0`), hardcoded to
`job_status.id = 1` ("Received", per `seed_bu_data.py:88`), dated from `job.created_at`, shown
only when no real `job_transaction` row with `status_id = 1` exists yet. `job-details-modal.tsx`
(the component rendering this, lines ~816-838) renders `t.status_name` verbatim with no
client-side special-casing.

**Fix (server-side, single source of truth):** in the synthetic branch of the `UNION ALL`
(`sql_store.py` ~line 4435-4451), change:
```sql
js.name AS status_name,
```
to:
```sql
CASE WHEN j.is_opening_job THEN 'Opening Job' ELSE js.name END AS status_name,
```
`j.is_opening_job` is already joined via the `FROM job j` in that branch тАФ no new join needed.
This only affects the synthetic `id=0` row; real `job_transaction` rows (including a later
legitimate transition back to "Received") are untouched, so subsequent history entries are
unaffected.

**Known pre-existing quirk, flagging (not fixing here тАФ out of tran.md's scope):** the synthetic
row is hardcoded to `job_status.id = 1` regardless of the job's actual initial `job_status_id`.
Opening Jobs lets the user pick any non-delivered status at creation (RECEIVED, ASSIGNED,
ESTIMATED, IN_PROGRESS, etc. тАФ `opening-job-form.tsx:366-381`), so today the transaction summary's
first entry always reads "Received" even when the job didn't start there. This plan's fix
(relabeling to "Opening Job") happens to paper over that mislabeling for opening jobs
specifically, but the underlying quirk (no real initial-state transaction row for *any* newly
created job) remains. Not proposing a fix for this since it's a bigger change (would mean
inserting a real `job_transaction` row at creation time, touching the shared
`resolve_create_single_job_helper` used by both Single Job and Opening Job) and isn't requested
by tran.md тАФ mentioning it here in case the user wants it addressed separately.
